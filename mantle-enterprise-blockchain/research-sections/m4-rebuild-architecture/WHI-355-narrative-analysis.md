# WHI-355: Mantle 业务叙事深度分析与区块链特性需求映射

> **Issue**: WHI-355 — Mantle 业务叙事深度分析与区块链特性需求映射  
> **Milestone**: M4: 叙事驱动分析与推倒重建的理想企业级方案  
> **Date**: 2026-05-07  
> **Status**: In Review  
> **Dependencies**: M1 全部研究 (WHI-334 ~ WHI-342), M2 全部对比 (WHI-343 ~ WHI-347)

---

## 执行摘要

本文档是 M4"推倒重建"阶段的基石分析。通过拆解 Mantle 四大核心业务叙事——**RWA 代币化、xstocks（代币化股票）、Payment（稳定币支付）和 DeFi 基础设施**——我们将每个叙事的完整业务生命周期映射到具体的区块链基础设施需求，并从跨叙事综合视角提炼出共性需求（核心基础设施）与差异化需求（可插拔模块），最终反推组件设计方向。

**核心发现**：

1. **隐私、准入控制和合规审计是所有叙事共需的核心能力**——这三项构成 Mantle 企业级改造的基础底座。
2. **叙事间存在显著的性能-隐私冲突**——Payment 要求 >10,000 TPS + <1s 延迟，而 RWA/xstocks 要求子交易级隐私 + 强合规，两者难以在同一执行环境中同时满足。
3. **多层架构不可避免**——类似 Tempo L1 + Zones 的分区模型，Mantle 需要至少两层：**公共层**（DeFi + 流动性）和**受管层**（RWA/xstocks/Payment 的隐私合规空间）。
4. **EVM 兼容性是不可放弃的战略优势**——M1 行业调研 (WHI-342) 已确认 EVM 赢得了标准之战，Canton 的 Daml 生态之小是其最大战略风险。

## 1. RWA 代币化叙事分析

### 1.1 业务场景拆解

RWA 是 Mantle 最复杂、合规要求最高的叙事，其完整生命周期涵盖五个阶段：

**资产发行阶段**

| 资产类别 | 发行特征 | 典型参与者 |
|---------|---------|-----------|
| 房地产 REIT | 大额低频，需持续评估，可产生租金收入 | SPV、托管人、审计师、评估师 |
| 公司债 | 标准化票面，固定/浮动利率，定期付息 | 发行方、承销商、评级机构、受托人 |
| 供应链应收账款 | 短期高频，基于贸易信用 | 核心企业、供应商、保理公司 |
| 碳信用额 | 需验证/认证，跨境流通 | 项目方、核证机构、交易平台 |
| 艺术品/收藏品 | 非标、需评估、可分割 | 艺术品托管方、鉴定机构 |

**一级市场**：合格投资者验证 → 认购 → 分配 → 锁定期管理。关键要求：只有经过 KYC/KYB 验证且满足特定条件（如净资产、投资经验）的投资者才能参与。认购过程需隐私保护（投资者不希望其他投资者看到自己的认购规模）。

**二级市场**：交易前合规检查（白名单验证、投资者适当性）→ 成交（原子结算或 DVP）→ 结算 → 交易后报告。二级市场的核心挑战是在保持合规控制的同时实现高效的交易匹配和结算。

**资产生命周期管理**：分红/付息 → 赎回 → 到期 → 清算。这些事件需要链上自动触发，同时保持与链下法律义务的一致性。

**跨境场景**：多司法管辖区合规（美国 Reg D/S、欧盟 MiCA、新加坡 MAS）、多币种结算（USD/EUR/SGD 稳定币）。

### 1.2 区块链特性需求矩阵

| 特性维度 | 需求等级 | 具体需求描述 | 参考案例 |
|---------|:-------:|------------|---------|
| **隐私** | 5 | **关键需求**。交易对手信息（谁买了什么）、持仓数量、资产估值必须私密。DVP 交易中，银行只需看到 IOU 部分，不需看到资产部分——需**子交易级隐私**。Canton 的 Merkle DAG 投影模型是最佳参考：DvP 中 Alice/Bob/Bank/Registrar 各自只看到相关子树 [WHI-335 §2.1.3]。 | Canton: Goldman Sachs DVP, $2T+/月处理量 |
| **准入控制** | 5 | **关键需求**。合格投资者准入（Reg D 仅限 accredited investors）、发行方准入（需持牌）、交易对手白名单。需支持多层准入：发行层（谁可以发行？）、投资层（谁可以购买？）、交易层（谁可以与谁交易？）。Tempo TIP-403 的复合策略（sender/recipient 分离）最适配 [WHI-340 §3.5]。 | Prividium RBAC 6 种权限类型; Tempo TIP-403 whitelist |
| **合规** | 5 | **关键需求**。证券法合规（美国 SEC Reg D/S/A+；欧盟 MiCA）、反洗钱/反恐融资（AML/CFT）、制裁筛查。需支持交易前合规检查（pre-trade）和交易后报告（post-trade）。Prividium 的 ZK 驱动无 PII 制裁筛查是行业创新方向 [WHI-338 §4.1]。 | Prividium ZK 制裁筛查; Canton Observer 模式 |
| **终局性** | 4 | DVP（券款对付）需要即时确定性终局性——交易一旦确认不可撤销。Canton 的 2PC 确认后即终局 [WHI-335 §3.4] 和 Prividium 的 1 秒 ZK 终局性 [WHI-337 §3.8] 都满足此需求。7 天挑战期（Optimistic Rollup）**完全不可接受**。 | Canton 确定性终局; Prividium 1s ZK 终局 |
| **性能** | 2 | 低性能要求。RWA 交易量有限（日均数百到数千笔），延迟容忍度高（秒级可接受）。瓶颈不在 TPS 而在合规检查复杂度。 | Canton 无公开 TPS 基准但处理 $2T+/月 |
| **互操作性** | 4 | 需与传统 CSD/DTCC 交互（SWIFT ISO 20022 消息格式）、跨链结算（RWA 资产在链 A 发行，在链 B 交易）。Canton 的多 Synchronizer 架构 [WHI-335 §1.4] 和 Prividium 的 ZKsync Connect [WHI-338 §1.3] 提供参考。 | Canton Global Synchronizer; SWIFT + Chainlink CCIP |
| **身份管理** | 5 | **关键需求**。KYC/KYB 要求、认证投资者身份验证、跨平台身份互认。需原生身份层——非外挂。Prividium 的链级 SSO 集成 [WHI-338 §3.1] 和 Tempo 的 AccountKeychain [WHI-339 §3.4] 均有参考价值。 | Prividium Okta SSO; Tempo AccountKeychain |
| **审计** | 5 | **关键需求**。监管报告（定期向 SEC/MAS/BaFin 提交）、完整审计追踪（从发行到赎回的完整生命周期）。Canton 的 Observer 模式 [WHI-334 §3.5.1] 让监管者直接看到子交易级数据，是最成熟的方案。 | Canton Observer; Prividium Private Explorer |
| **数据主权** | 4 | 资产数据归属需明确（谁"拥有"链上的资产记录？）。跨境数据合规（GDPR 要求数据可删除，美国 SEC 要求数据保留 5-7 年）。Canton 的虚拟全局账本 [WHI-335 §1.1] 天然满足数据主权——每方只存自己的投影。Prividium 的 Validium 链下 DA [WHI-338 §1.1] 也满足。 | Canton 分布式投影; Prividium Validium 链下 DA |

### 1.3 参考案例分析

**Canton Network (Goldman Sachs/BNY Mellon)** — 最直接的 RWA 参考。450+ 参与者，$2T+/月处理量。核心优势是子交易级隐私和 Observer 监管模式。局限：非 EVM，Daml 生态小 [WHI-334, WHI-335]。

**Prividium (Cari Network 银行代币化存款)** — 5 家美国银行 ($600B+ 存款) 选择 Prividium 而非 Canton 的关键原因是"数学信任 vs 协议信任"。银行不愿在竞争对手的基础设施上构建 [WHI-337 §4.1]。

**Tempo TIP-20 + TIP-403** — 合规代币框架。TIP-403 的 precompile 级强制执行是 EVM 生态中最彻底的合规方案——每笔 TIP-20 transfer 自动调用 `transferAuthorized()` [WHI-340 §3.5]。

**Ondo Finance / Securitize / Centrifuge** — 公链 DeFi RWA 协议。展示了 EVM 生态中 RWA 发行的可行性，但缺乏原生隐私和机构级合规。

## 2. xstocks（代币化股票）叙事分析

### 2.1 业务场景拆解

xstocks 将上市公司股票 1:1 映射为链上 token，实现 7×24 交易和 T+0 结算。这是对传统证券市场的颠覆性改造。

**代币化股票发行**：1:1 映射上市公司股票，需要链下托管商持有实物股票，链上发行等量 token。涉及 SEC 或等效监管机构的审批。

**交易场所设计**：

| 场景 | 要求 | 隐私需求 |
|------|------|---------|
| 7×24 交易 | 传统市场闭市时段仍可交易 | 中等：价格公开，持仓可私密 |
| 盘前/盘后市场 | 扩展交易时段，流动性较低 | 高：大单信息泄露可能移动价格 |
| 暗池交易 | 大宗交易不在公开订单簿显示 | **极高**：交易双方身份、价格、数量均需私密 |
| 做市商 | 提供流动性，需知道订单流 | 高：做市策略需保密 |

**结算**：T+0 原子结算（vs 传统 T+2），DVP 同步交割。原子结算消除了对手方风险，但要求链上有即时终局性。

**公司行为处理**：分红（链上 token 持有者自动获得分红）、拆股（链上 token 数量按比例调整）、配股、投票（链上 token 赋予投票权）。

**跨境交易**：美股 token 在全球 24 小时交易，需处理跨境资本流动合规和多币种结算。

### 2.2 区块链特性需求矩阵

| 特性维度 | 需求等级 | 具体需求描述 | 参考案例 |
|---------|:-------:|------------|---------|
| **隐私** | 5 | **关键需求**，尤其是暗池场景。需支持：(1) 持仓隐私——不暴露大户持仓；(2) 交易隐私——暗池交易完全不可见；(3) 大单保护——防止信息泄露导致的价格操纵。Canton 的子交易级隐私最适合暗池场景 [WHI-335 §2.1]。Tempo Zones 的单序列器隐私适合交易执行隔离 [WHI-340 §7.2]。 | Canton 暗池 DVP; Tempo Zone 隔离 |
| **准入控制** | 5 | 券商/经纪人准入（需持牌）、投资者适当性检查（不同证券有不同投资者要求）。需与传统券商的准入体系对接。 | Prividium RBAC + SSO; Tempo TIP-403 |
| **合规** | 5 | SEC 规则（Reg NMS——价格保护、Reg SHO——卖空限制、Regulation ATS——替代交易系统）、跨境证券法、交易报告义务。 | 传统交易所合规框架 |
| **终局性** | 5 | **最高要求**。T+0 原子结算需要即时确定性终局性。交易确认后立即可用于后续交易——不接受任何等待期。 | Canton 2PC 终局; Prividium 1s ZK 终局 |
| **性能** | 4 | 高于 RWA。高频交易（HFT）场景要求低延迟（<100ms）；订单簿交易需要高吞吐（峰值 >5,000 TPS）。但不如 Payment 极端。 | Tempo 600ms 出块; Prividium >15K TPS |
| **互操作性** | 4 | 与传统券商/清算所的集成（FIX 协议适配）、跨链股票 token 流通。 | DTCC DLT 探索; SIX 代币化试点 |
| **审计** | 5 | **关键需求**。交易所监控（market surveillance）——检测内幕交易、操纵行为；异常交易报告（SAR/STR）；完整的订单到结算审计追踪。 | Canton Observer 实时审计 |
| **数据主权** | 3 | 中等需求。股票数据本身有公共属性（价格、交易量），但持仓和订单流数据高度敏感。 | — |

### 2.3 参考案例分析

**Securitize + BlackRock BUIDL Fund** — 最大的代币化基金 ($500M+)，证明了链上证券的市场接受度。但运行在公链上，缺乏机构级隐私。

**Backed Finance** — 欧洲代币化 ETF，1:1 映射 ETF，受 BaFin 监管。展示了 MiCA 框架下代币化证券的合规路径。

**DTCC DLT 探索** — 传统清算所探索区块链清结算，验证了 T+0 结算的行业需求。

**SIX / JPX / SGX 交易所试点** — 传统交易所的代币化试点，验证了 7×24 交易和原子结算的可行性。SIX Digital Exchange (SDX) 已投入生产。

## 3. Payment / 稳定币支付叙事分析

### 3.1 业务场景拆解

Payment 叙事对性能要求最高，但合规粒度与 RWA/xstocks 不同。

**B2B 跨境支付**：企业间大额跨境转账（$10K-$100M 级别），当前通过 SWIFT 需 1-5 天，区块链可实现秒级。参与者：付款银行、收款银行、外汇提供商、合规筛查服务。

**B2C 消费支付**：日常消费级支付（$1-$10K），要求类 Visa/Mastercard 体验（<2s 终局、极低费用）。参与者：消费者、商户、支付处理商、发卡方。

**企业资金管理**：企业内部多币种流动性池管理、净额结算、内部银行。TCMAG 12 条原则 [WHI-337 §4.4] 定义了代币化企业现金管理的标准。

**DeFi 支付集成**：链上支付与 DeFi 收益的组合——闲置支付资金可自动存入收益协议。这需要公共 DeFi 和受管支付通道之间的桥接。

### 3.2 区块链特性需求矩阵

| 特性维度 | 需求等级 | 具体需求描述 | 参考案例 |
|---------|:-------:|------------|---------|
| **隐私** | 4 | 交易金额和对手方隐私——企业不希望竞争对手看到其支付流向和规模。商业竞争信息保护。B2C 场景相对宽松（仅对非参与方私密）。Tempo Zone 的加密存款 (ECIES) [WHI-340 §5.1] 和消毒区块数据 [WHI-340 §7.2] 是最直接的参考。 | Tempo Zone ECIES 加密存款; Prividium Validium |
| **准入控制** | 4 | VASP（Virtual Asset Service Provider）注册、发卡方/收单方准入。B2B 场景需严格准入（银行间只允许已知对手方）；B2C 场景可宽松（KYC 即可）。 | Tempo TIP-403; Prividium Proxy RPC |
| **合规** | 4 | Travel Rule（$3,000+ 转账需记录发送方/接收方信息）、AML/CFT、制裁筛查。注意：支付场景的合规要求与证券场景不同——更侧重反洗钱而非投资者保护。 | Tempo Zone Sequencer 合规检查; Partior 银行间合规 |
| **终局性** | 5 | **最高要求**。B2C 消费支付需要**亚秒级终局性**——用户扫码后立即确认。B2B 跨境支付需要秒级终局性。 | Tempo 600ms Simplex BFT; Prividium 1s ZK 终局 |
| **性能 (TPS)** | 5 | **最高要求**。B2C 消费级需 Visa 量级（峰值 ~65K TPS，日均 ~2K TPS）。至少需要 >10,000 TPS 以应对峰值。Tempo 的 Payment Lane 保证了支付交易的确定性吞吐量 [WHI-339 §2.2]——这是最直接的参考。 | Tempo Payment Lane; Prividium >15K TPS (Atlas) |
| **费用** | 5 | **关键需求**。每笔交易成本需极低（<$0.001），尤其是 B2C 微支付场景。当前链上 gas 成本对小额支付不可接受。Tempo 的 stablecoin 计价 gas [WHI-339 §2.4] 和 Prividium 的 <$0.0001/tx [WHI-337 §3.8] 提供了可行路径。 | Tempo 稳定币 gas; Prividium <$0.0001 |
| **互操作性** | 4 | 与 SWIFT/银行网络的互操作、跨链支付（USDC 在链 A → USDT 在链 B）。Circle CCTP 和 LayerZero 是参考。 | Partior 银行间; Circle CCTP; SWIFT + Chainlink |
| **多币种** | 4 | 多种稳定币支持（USDC、USDT、EURC 等）、自动兑换（FX on-chain）。Tempo TIP-20 的原生多 token precompile 最适配 [WHI-339 §2.4]。 | Tempo TIP-20 多 token; Partior 多币种 |

### 3.3 参考案例分析

**Tempo Payment Lane + TIP-20 + Zone 隐私支付** — **最直接的参考**。Payment Lane 保证了支付交易不受一般链拥堵影响 [WHI-339 §2.2]；TIP-20 是 precompile 级别的原生稳定币标准 [WHI-339 §2.4]；Zone 提供隐私支付空间 [WHI-339 §3.2]。

**Partior (JPMorgan/DBS/Temasek)** — 银行间支付网络，验证了区块链在跨境大额支付中的可行性。许可制网络，受监管银行参与。

**Circle CCTP** — 跨链 USDC 协议，展示了链间稳定币互操作性的标准模式。

**Stripe 稳定币支付集成** — Stripe 接入 USDC 支付，验证了稳定币在 B2C 商业场景中的产品市场契合度。

## 4. 其他潜在叙事

### 4.1 DeFi 基础设施

| 子场景 | 特殊需求 | 与主叙事的关系 |
|--------|---------|---------------|
| DEX (去中心化交易) | 无许可可组合性、MEV 保护、高吞吐 | 为 RWA/xstocks 提供二级市场流动性 |
| 借贷协议 | 清算机制、oracle 集成、风控 | RWA token 可作为抵押品 |
| 收益协议 | 自动化策略、时间锁 | Payment 闲置资金的收益来源 |
| 保险 | 链上理赔、参数化保险 | 为 RWA 资产提供保险保障 |

**DeFi 的核心矛盾**：DeFi 的价值来自无许可可组合性（任何协议可调用任何协议），而企业叙事要求许可和合规。两者需要在不同层共存——公共 DeFi 在基础层，企业场景在受管层。

### 4.2 供应链金融

多层级参与者（核心企业 → 一级供应商 → 二级供应商 → ...)，需要部分信息共享（每层只看到直接相邻层的信息）。Canton 的子交易级隐私天然适合此场景。

### 4.3 身份与信用

链上信用评分、可验证凭证 (Verifiable Credentials)。为 RWA 投资者准入和 Payment KYC 提供基础设施。Tempo 的 AccountKeychain [WHI-339 §3.4] 和 Prividium 的 SSO 集成 [WHI-338 §3.1] 可作为身份层参考。

### 4.4 企业 DAO 治理

企业级 DAO 的投票和决策机制——用于 RWA 资产的持有人投票、xstocks 的股东投票。需支持隐私投票（不暴露个人投票偏好）和加权投票（按持有量加权）。

### 5.2 共性需求 vs 差异化需求

#### 所有叙事共需的核心能力 → 核心基础设施

| 共性需求 | 分析 | 设计含义 |
|---------|------|---------|
| **准入控制** | RWA(5) + xstocks(5) + Payment(4) + 供应链(3) 均需许可参与。DeFi 除外。 | 必须在**协议层**实现，不能仅靠合约层。参考 Tempo TIP-403 precompile（自动检查、不可绕过）+ Prividium Proxy RPC（网关级过滤）。 |
| **合规审计** | RWA/xstocks(5) + Payment(4) + 供应链(3) 均需监管报告和审计追踪。 | 需要**原生审计框架**——Canton Observer 模式（监管方作为参与者直接看到数据）+ Prividium Private Explorer（角色化审计视图）的组合。 |
| **身份管理** | RWA(5) + xstocks(4) + Payment(4) + 供应链(3) 均需 KYC/KYB。 | 需要**原生身份预编译**——将 KYC 状态、投资者资质编码在协议层。参考 Tempo AccountKeychain + Prividium SSO 集成。 |
| **终局性** | 所有叙事至少需要 3+，xstocks/Payment 需要 5。 | **确定性即时终局性**是底线要求。Optimistic Rollup 的 7 天挑战期完全不可接受——必须引入 ZK 证明（如 Prividium）或 BFT 终局（如 Canton/Tempo）。 |
| **隐私** | RWA/xstocks(5) + Payment/供应链(4)。DeFi(2) 除外。 | 需要**多级隐私框架**——子交易级隐私（RWA DVP）+ 链级隐私（Payment Zone）+ 选择性披露（合规审计）。 |
| **互操作性** | RWA/xstocks/Payment(4) + DeFi/供应链(3)。 | 需要**多层互操作**——与以太坊 L1 的结算连接 + 受管域之间的跨域通信 + 与传统金融系统的集成（SWIFT/FIX）。 |

#### 叙事特定的差异化需求 → 可插拔模块

| 差异化需求 | 来源叙事 | 设计含义 |
|-----------|---------|---------|
| **超高 TPS (>10K)** | Payment | 需要专用执行通道——参考 Tempo Payment Lane（blockspace 分区保证支付吞吐量不受通用交易影响 [WHI-339 §2.2]） |
| **极低费用 (<$0.001)** | Payment | 需要 stablecoin 计价 gas + 费用赞助机制。参考 Tempo 的 attodollar 计价 [WHI-339 §2.4] 和原生 fee sponsorship [WHI-339 §2.3] |
| **暗池交易** | xstocks | 需要专用隐私执行环境——隔离的 Zone 或 Domain，只有做市商和交易方可见 |
| **公司行为处理** | xstocks | 链上自动执行分红、拆股等公司行为——需要特定的智能合约框架 |
| **多币种 FX** | Payment | 链上外汇兑换——需要原生多 token 支持和 AMM/订单簿 |
| **供应链多层隐私** | 供应链 | 每层只看到直接相邻层——需要 Canton 式子交易级隐私 |

#### 叙事间的冲突与张力

| 冲突 | 分析 | 解决方向 |
|------|------|---------|
| **高 TPS vs 强隐私** | Payment 需 >10K TPS 但 RWA 需子交易级隐私。加密操作（ECIES、ZK 证明）增加计算开销，降低吞吐。 | **物理分离**：Payment 在高性能低隐私通道执行，RWA 在低性能高隐私域执行。两者通过跨域桥接连接。 |
| **无许可可组合性 vs 许可准入** | DeFi 需要任何协议可调用任何协议；RWA/xstocks 需要白名单准入。 | **层次分离**：公共 DeFi 在基础层（无许可），企业叙事在受管层（许可）。受管层可选择性接入基础层流动性。 |
| **数据公开 vs 数据隐私** | DeFi 需要全局状态可查（TVL、价格），RWA 需要交易隐私。 | **选择性披露**：聚合指标（TVL、总供应量）可公开，个体交易数据私密。参考 Prividium 的可配置公共端点 [WHI-338 §1.3]。 |

### 5.3 组件设计需求映射

从特性需求反推组件设计，参考 M1/M2 研究中的最佳实践：

| 共性特性 | 组件设计 | 参考实现 | 设计理由 |
|---------|---------|---------|---------|
| **准入控制** | **协议层策略注册表 (Policy Registry)** — precompile 实现，支持 whitelist/blacklist/compound 策略，token transfer 时自动检查 | Tempo TIP-403 [WHI-340 §3.5] 的 precompile 级强制执行 + Prividium Proxy RPC [WHI-338 §3.2] 的网关层过滤 | TIP-403 是 EVM 生态中最彻底的合规方案——precompile 级自动检查不可绕过 [WHI-344 §1.1]。Proxy RPC 提供额外的网关层纵深防御。 |
| **合规审计** | **链上合规框架 + 审计追踪** — Observer/Auditor 角色可被授权查看特定范围的交易数据；ZK 证明支持无 PII 合规验证 | Canton Observer 模式 [WHI-334 §3.5.1] 的实时数据访问 + Prividium 选择性披露 [WHI-338 §4.1] 的 5 种机制 | Canton 的 Observer 模式最成熟（$2T+/月验证），但需适配 EVM 环境。Prividium 的 ZK 合规证明是面向未来的方向。 |
| **身份管理** | **原生身份预编译 (Identity Precompile)** — 存储 KYC 状态、投资者资质、角色信息；支持跨平台身份互认；集成企业 SSO | Tempo AccountKeychain [WHI-339 §3.4] 的 root/access key 委托 + Prividium SSO [WHI-338 §3.1] 的 Okta OIDC 集成 | AccountKeychain 的 P256/WebAuthn 原生支持是面向消费者的创新；SSO 集成是面向企业的必需。两者需组合。 |
| **终局性** | **确定性终局性引擎** — BFT 共识或 ZK 有效性证明，交易确认后不可回滚 | Canton 2PC [WHI-335 §3.1] 的确定性终局 + Prividium ZK 终局 [WHI-337 §3.8] 的 1 秒终局 | Optimistic Rollup 的挑战期对企业场景完全不可行。必须引入 ZK 证明（数学保证）或 BFT 共识（阈值保证）。 |
| **隐私** | **多级隐私框架** — (1) 链级隐私（Validium 链下 DA）+ (2) 域级隐私（受管域内部状态对外不可见）+ (3) 子交易级隐私（多方交易中各方只看到相关部分）+ (4) 选择性披露 | Canton Merkle DAG 投影 [WHI-335 §2.1] + Prividium Validium [WHI-338 §1.1] + Tempo Zone 隔离 [WHI-340 §7.2] | 三种隐私范式各有最佳适用场景 [WHI-343 §1]。RWA DVP 需 Canton 级子交易隐私；Payment 需 Tempo 级域隔离；对外结算需 Prividium 级 ZK 证明。 |
| **支付通道** | **专用支付执行通道 (Payment Lane)** — blockspace 分区保证支付交易的确定性吞吐量和延迟 | Tempo Payment Lane [WHI-339 §2.2] 的 gas 分区机制（`general_gas_limit` vs `shared_gas_limit`） | Payment Lane 是 Tempo 最独特的创新——payload 检查即可分类，无需状态访问。"反噪声邻居"保证 [WHI-339 §2.2]。 |
| **互操作性** | **多层桥接架构** — (1) 与 Ethereum L1 的结算桥（ZK 证明 or Optimistic）+ (2) 受管域间的跨域通信 + (3) 与传统系统的 API 网关 | Canton Global Synchronizer [WHI-334 §3.7] + Prividium ZKsync Connect [WHI-338 §1.3] + Tempo ZonePortal [WHI-340 §3.4] | 企业互操作性有四个维度 [WHI-347 §1]，每个维度需要不同的桥接机制。单一桥接方案无法满足所有需求。 |

### 5.4 系统整体运作模型

基于上述分析，Mantle 企业级区块链需要一个**多层多域架构**：

                │                                               │

          │ RWA/xstocks    │ │ Payment      │ │ 供应链金融      │

#### 5.4.1 哪些叙事可以在同一链上共存？

- **DeFi + 公共层**：无许可 DeFi 协议在 Mantle 公共基础层运行，任何人可参与。
- **RWA + xstocks**：可以在同一受管域中共存——两者的合规框架（证券法）和隐私需求（DVP 隐私）高度重叠。
- **Payment**：必须有独立的执行通道——其性能需求（>10K TPS, <1s）与 RWA/xstocks 的隐私计算开销不兼容。

#### 5.4.2 是否需要多链架构？

**是的。** 类似 Tempo L1 + Zones 的分层模型是必要的，原因：

1. **性能-隐私冲突**：Payment 的 >10K TPS 与 RWA 的子交易级隐私无法在同一执行环境中同时满足。
2. **合规隔离**：不同叙事受不同监管框架约束（SEC vs MiCA vs PSD2），需要独立的合规配置。
3. **数据主权**：不同机构的数据不应混在同一状态空间中。
4. **治理独立性**：不同受管域可以有不同的治理模型（不同的策略管理员、不同的验证者集合）。

#### 5.4.3 公共 DeFi 和企业场景如何共存？

**桥接模式**：受管域中的 RWA token 可以通过合规桥接（带准入检查）进入公共层的 DeFi 协议。例如：

1. 合格投资者在受管域中持有 RWA token
2. 通过合规桥（白名单检查）将 token 映射到公共层
3. 在公共层 DEX 进行二级市场交易
4. 交易结果通过合规桥回到受管域更新持仓

关键设计：桥接时必须检查接收方是否在准入白名单中——非白名单地址不能持有受管资产。

#### 5.4.4 治理和升级模型

- **公共基础层**：社区治理（类似现有 Mantle 治理），协议升级需 DAO 投票。
- **企业共享层**：联盟治理——策略注册表的更新需要授权管理员操作。
- **受管域**：域运营商治理——每个域可独立配置合规策略、准入规则、性能参数。

## 7. 结论与 M4 后续方向

本分析确立了三个指导 M4 所有后续设计决策的核心结论：

**结论 1: 多层多域架构是唯一可行的方案。** 四大叙事的需求差异过大——Payment 的性能需求与 RWA 的隐私需求不可能在同一执行环境中同时满足。Mantle 需要：公共基础层（DeFi + 流动性）→ 企业共享层（准入控制 + 身份 + 合规）→ 受管域（叙事特定的隐私和性能配置）。

**结论 2: 五个核心组件构成改造底座。** 策略注册表（TIP-403 式 precompile）+ 身份预编译（AccountKeychain + SSO）+ 合规审计框架（Observer + 选择性披露）+ 确定性终局性引擎（ZK 证明或 BFT）+ 多级隐私框架。这五个组件是所有叙事的共性需求，必须在协议层实现。

**结论 3: EVM 兼容性 + 以太坊锚定是不可放弃的战略优势。** M1 行业调研已确认 EVM 赢得了标准之战。Canton 的 Daml 生态之小是其最大战略风险。Prividium 的"以太坊锚定 + EVM 兼容 + 企业隐私"组合代表了行业趋势——Mantle 应沿这一方向演进，而非切换到非 EVM 技术栈。
